## 2. The Product Thinking

**Runner-up venues:** The "Also Considered" panel does something subtle — it makes the winning venue feel *chosen*, not just returned. When you see that Webster's Bookstore Café scored 63 and lost on vibe match, the Elixr pick feels more defensible. You understand the tradeoff. This is especially useful in a Ditto AI context because it surfaces the scoring system's reasoning without requiring the user to ask for it.

The "Lost on: X" copy is deliberately minimal. It doesn't say "Webster's scored lower on vibe alignment than Elixr." It says "Lost on: vibe match (8/25)." Short enough to scan, specific enough to mean something.

**Compatibility banner:** The matching engine already tells you which *venue* fits the pair. The compatibility engine tells you whether the *people* fit each other. These are different signals. A high venue score means this location suits their combined profile. A high compatibility score means their underlying personalities align. Maya and Alex score 92/100 compatible — same energy level, overlapping interest groups (music, reading), shared values around curiosity and authenticity, identical date preference. That's a different and more personal statement than "Elixr Coffee scored 72."

Putting the banner first in the layout is intentional. The person-to-person signal is the most emotionally resonant piece of information on the page. Leading with "Highly Compatible: 92/100" before showing venue details sets the right frame.

## 3. The Decisions We Made (and What We Rejected)

**Decision: `biggestGapDimension` instead of `weakestDimension` for "Lost on" copy**

The first implementation used `weakestDimension` — sorted the runner-up's breakdown ascending, picked the lowest value. This has an edge case: if the winning venue scored the same (or lower) on that dimension, the "Lost on" label is misleading. Example: Elixr scores 8/25 on activity. The Tavern also scores 8/25 on activity. Showing "Lost on: activity fit (8/25)" for The Tavern implies it underperformed on activity when really it tied.

The fix was `biggestGapDimension(winnerBreakdown, runnerBreakdown)` — for each dimension, compute `winner[dim] - runner[dim]`, return the dimension with the largest positive gap. That's the actual reason the runner-up lost relative to the winner. The fallback (if the runner-up ties or beats the winner on every dimension, which is practically impossible given a lower total score) shows the absolute weakest dimension.

**Decision: compatibility engine is deterministic, not LLM-generated**

The compatibility score could have been generated by sending both persona profiles to Claude and asking for a compatibility analysis. Rejected for three reasons. First, latency — the card generation subprocess already adds 5–10s to the response; adding a second LLM call would make the UI feel sluggish. Second, reproducibility — the same pair should always get the same compatibility score. A deterministic engine makes the number trustworthy and debuggable. Third, interview positioning — the two-stage architecture (deterministic scorer + LLM for narrative) was already the system's core design principle. Extending that principle to person compatibility reinforces the pattern rather than breaking it.

**Decision: group-based interest and values matching, not exact string matching**

`"indie music"` and `"live music"` are different strings but compatible interests. `"curiosity"` and `"intellectual curiosity"` express the same value. Exact string matching would score them as non-overlapping. The engine uses `INTEREST_GROUPS` and `VALUES_GROUPS` — sets of semantically related terms — and awards points per group where *both* personas have at least one member. This makes the scoring robust to the way interests are phrased in the persona data without requiring normalization or embeddings.

**Decision: render CompatibilityBanner first, venue card second**

The venue card was first in the original layout. Moving compatibility above it is a deliberate product decision: in a dating context, "you two are Highly Compatible" is a warmer, more human-feeling opening than "here's a coffee shop." The venue card is still prominent — it's the actionable output — but the compatibility banner sets the emotional context for reading it.

**Rejected: showing compatibility score breakdown in the "Also Considered" panel**

A version of the UI could show why each runner-up *venue* is less compatible with the pair's compatibility profile. Rejected as too much information. The panel is meant to be scannable — venue name, score, one reason it lost. Adding a second layer of breakdown would clutter it.

## 5. Anticipated Interview Questions + Answers

**"How does the compatibility score work?"**
Four dimensions, each 0–25. Energy alignment is `max(0, 25 - abs(a.energy - b.energy) * 6)` — a 1-point difference costs 6 points, 4+ points apart floors at 1. Interest and values overlap are group-based: I defined clusters of related terms and award `5 + groups_with_both * 8` points per overlap, capped at 25. Vibe compatibility is a lookup table — identical date preferences score 25; divergent pairs like "low_pressure" + "social" score 8. The group-based approach means "indie music" and "live music" correctly register as a shared interest group even though they're different strings.

**"Why show runner-up venues?"**
Two reasons. First, it makes the winning pick feel earned — you can see what it beat and why. Second, it surfaces the scoring system's reasoning without the user having to ask. "Lost on: vibe match (8/25)" is a more specific and trustworthy explanation than "this venue scored highest." It also opens a natural demo conversation: "what if they actually wanted a livelier venue?" — you can point at the runner-up and show the tradeoff.

**"Why is the 'Lost on' label computed relative to the winner rather than absolute?"**
If a runner-up scores 8/25 on activity and the winning venue also scores 8/25 on activity, showing "Lost on: activity fit" is misleading — they tied on that dimension. The runner-up lost because the winner scored higher on *other* dimensions. Computing `winner[dim] - runner[dim]` and taking the max gap gives you the actual competitive differentiator, not just the runner-up's weakest absolute score.

**"Why is the compatibility engine deterministic? Couldn't an LLM write a better compatibility narrative?"**
An LLM could write a more nuanced narrative, yes. But the score needs to be stable — the same pair should always get the same number. If the compatibility score is LLM-generated, it drifts between sessions and you can't unit test it. The two-stage pattern is intentional throughout this system: deterministic scoring for the things that need to be auditable and reproducible, LLM for the things that genuinely need language and empathy. Compatibility scoring falls in the first category. The card narrative falls in the second.

## 6. The Sentiment

The runner-up panel was the most satisfying addition because the data was already there. `rank_venues_for_pair()` returns a sorted list on every request — positions 1 and 2 were being thrown away. Stopping that discard took one line in `main.py` and the rendering was straightforward. The "Lost on" copy took more thought: the first version was technically correct but had the edge case where you'd show a misleading label when the winner and runner-up tied on the weakest dimension. Catching that and fixing it with the gap-based approach made the output genuinely more accurate, not just superficially different.

The compatibility engine was satisfying for a different reason — it's the first feature that takes the persona data and uses it to say something *about the people* rather than about the venues. The scoring logic is simple (a few lookup tables and group membership checks) but the output feels meaningful because it's grounded in real profile dimensions. Maya and Alex scoring 92/100 isn't a coincidence of phrasing — they have the same energy level, share two interest groups, share two values groups, and have identical date preferences. The number reflects something real about the data.
